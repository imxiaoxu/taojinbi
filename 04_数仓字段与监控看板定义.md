# 数仓字段与监控看板定义

## 1. 事实表：干预决策与执行

建议表名：`dwd_coin_game_agent_intervention_execution_di`

| 字段 | 类型 | 说明 |
|---|---|---|
| `dt` | date | 分区日期 |
| `event_time` | timestamp | 服务端事件时间 |
| `user_id_hash` | string | 脱敏用户 ID |
| `session_id` / `trace_id` / `decision_id` | string | 端到端关联标识 |
| `experiment_id` / `ab_group` | string | 实验与分组 |
| `scene_type` | string | 如 `level_2_block` |
| `trigger_event_name` | string | 如 `level_fail` |
| `game_variant` / `level_id` / `fail_count_same_level` | string/int | 触发时游戏状态 |
| `feature_version` / `workflow_version` | string | 可复现决策版本 |
| `plan_json` | string | 结构化计划快照 |
| `action_type` / `action_step` | string/int | 实际动作与顺序 |
| `execution_status` / `delivery_status` | string | 工具和前端展示结果 |
| `error_code` / `latency_ms` | string/int | 稳定性监控 |
| `idempotency_key` | string | 重复执行核验 |
| `feedback_type` | string | accept/reject/close/none |
| `reward_amount` / `cost_amount` | decimal | 成本字段；首期应为 0 |

## 2. 实验用户宽表

建议表名：`dws_coin_game_level2_agent_ab_user_7d`

| 字段 | 口径 |
|---|---|
| `user_id_hash`、`experiment_id`、`ab_group`、`enroll_time` | 首次满足入组规则时写入，不可覆盖 |
| `is_new_user`、`device_os`、`rules_viewed_before_fail` | 入组前特征快照，用于平衡性与异质性分析 |
| `level_2_pass_24h`、`game_complete_24h`、`level_2_exit_10m` | 主/过程结果 |
| `intervention_shown`、`intervention_accepted`、`action_success` | 执行漏斗 |
| `retention_7d`、`add_to_cart_7d`、`order_7d` | T+7 回填结果 |
| `reward_cost_7d`、`coupon_cost_7d` | 成本护栏 |

## 3. 监控看板

### 页面 A：实验健康度（实时/T+1）

| 指标 | 计算 | 告警 |
|---|---|---|
| 入组人数 | `count(distinct user_id_hash)` 按组 | 与计划分流偏差 > 5% |
| SRM p-value | 对 1:1:1 分流做卡方检验 | `p < 0.01` |
| 事件完整率 | 有 `level_fail` 的入组用户中，具备 `level_complete/game_exit` 回传的比例 | < 99% |
| Dify P95 / 工具失败率 | 决策/工具日志 | 超过方案阈值 |

### 页面 B：关卡与动作漏斗（实时/T+1）

`入组 -> 决策生成 -> 前端展示 -> 用户接受 -> Level 2 通过 -> 游戏完成`

按 `ab_group`、新老用户、Android/iOS、是否先查看规则拆分。必须同时展示每层用户数与相对上一层转化率，避免只展示百分比。

### 页面 C：效果与护栏（T+1/T+7）

| 指标 | 对比 | 判断 |
|---|---|---|
| `level_2_pass_rate` | Agent vs Holdout；Agent vs 固定规则 | 主效果 |
| `game_completion_rate` | 同上 | 次主效果 |
| `level_2_immediate_exit_rate` | 三组 | 体验护栏 |
| `retention_7d/add_to_cart_7d/order_7d` | 三组 | 延迟业务效果 |
| 成本、重复执行、负反馈 | 三组 | 风险与ROI |

### 页面 D：Agent 运行质量（实时/T+1）

展示场景命中率、计划长度分布、动作执行成功率、fallback 比例、错误码 Top N、同用户重复干预次数和决策版本差异。

## 4. 核心分析 SQL 口径（伪 SQL）

```sql
SELECT
  ab_group,
  COUNT(*) AS enrolled_users,
  AVG(level_2_pass_24h) AS level_2_pass_rate,
  AVG(game_complete_24h) AS game_completion_rate,
  AVG(level_2_exit_10m) AS immediate_exit_rate,
  AVG(retention_7d) AS retention_7d,
  AVG(order_7d) AS order_7d
FROM dws_coin_game_level2_agent_ab_user_7d
WHERE experiment_id = '${experiment_id}'
GROUP BY ab_group;
```

所有效果表需要同时输出 `n`、比例、差值、95% CI、p-value 和 SRM 结果；不只展示 uplift。

## 5. 监控异常排查与处置 Runbook

告警触发后先确认统计窗口和成熟队列，再沿 `前端事件 -> Backend -> Dify/策略 -> 业务工具 -> 数据回流` 顺序定位。任何修复均须保留原始 `trace_id`、错误码和版本信息，不得直接修改历史事实记录。

| 告警场景 | 首要核验 | 排查顺序 | 责任团队 | 临时处置 | 恢复标准 |
|---|---|---|---|---|---|
| 入组量突降或三组比例异常 | 页面流量、入组事件量、SRM p-value | 前端 `level_fail` 上报 -> Backend 去重/过滤 -> 实验分桶与固定桶 -> 数仓去重 | 前端、实验平台、数据 | 暂停扩量，不改变既有用户分桶 | 连续 2 个小时事件量恢复，SRM `p >= 0.01` |
| 前端干预展示率下降 | `decision_success` 与 `agent_intervention_shown` 差值 | Backend 响应 -> 前端 Schema 解析 -> 组件加载 -> 展示回执 | 前端、后端 | 返回静态提示；保留未展示决策日志 | 展示失败率 < 0.5%，连续 2 个监控窗口稳定 |
| Dify P95 超过 1.5 秒 | 请求量、超时率、模型与工作流版本 | Backend Queue -> Dify Proxy -> Workflow 节点耗时 -> 模型服务 | Agent、后端 | 触发 1.5 秒超时，切换固定静态兜底 | P95 <= 1.5 秒且超时率 < 1% |
| Dify 返回格式不合法 | Schema 校验错误码、缺失字段、版本号 | `workflow_version` -> 原始响应 -> JSON Schema -> Action 白名单 | Agent、后端 | 拒绝执行计划，返回 `show_static_hint` | 计划校验通过率 >= 99.5%，连续 2 个窗口无非法动作 |
| 工具失败率超过 1% | 按 `action_type/error_code` 拆分 | Backend Executor -> 幂等键 -> 业务接口状态 -> 权限/限流 | 后端、业务服务 | 非权益动作最多重试 1 次；权益动作写 `pending`，禁止重复发放 | 工具失败率 <= 1%，重复执行率为 0 |
| Level 2 通过率或完成率突降 | 分组样本量、观察窗、版本变更 | 数据完整性 -> 分流平衡 -> 策略版本 -> 游戏配置 -> 用户分层 | 数据、Agent、游戏产品 | 若较 Holdout 恶化 > 3pp 且连续两日成立，暂停 Agent 扩量 | 护栏恢复且根因复核通过 |
| 指标为 0、突增或组间完全一致 | 原始事件量、分区、去重键、成熟队列 | ODS 到 DWD -> 事实表 Join -> 宽表回填 -> BI 缓存与筛选 | 数据、BI | 看板标记“数据异常”，禁止业务解读 | 原始事件、事实表、宽表三层对账误差 < 0.5% |
| T+7 指标未更新 | 最后一位入组时间、成熟用户数、回填任务状态 | 成熟队列判定 -> 调度任务 -> 交易/留存 Join -> 分区发布 | 数据 | 展示“未成熟/延迟”，不以前一日值填充 | 回填完成且成熟用户数与入组台账一致 |
| fallback 比例持续升高 | 按原因拆分 `guardrail/system/tool` | 特征缺失 -> 频控 -> Agent 异常 -> 工具异常 | Agent、后端、数据 | 保持 Holdout 不变，Agent 组降级固定规则或静态提示 | fallback 恢复至基线范围，Top 原因完成复核 |

### 5.1 告警记录要求

每次告警必须记录 `alert_id`、`metric_name`、`detected_at`、`experiment_id`、`ab_group`、`workflow_version`、`feature_version`、`owner_team`、`root_cause`、`mitigation`、`resolved_at` 和 `postmortem_url`。同一异常跨多个指标时使用同一个 `incident_id` 关联，避免重复统计事故。

### 5.2 升级规则

- P0：合规事故、重复发奖、核心游戏不可用，立即关闭干预并通知负责人。
- P1：SRM、关键埋点缺失、计划格式错误或工具失败率越线，15 分钟内响应并暂停扩量。
- P2：单一分层效果波动或非核心数据延迟，工作日内完成复核，不影响主实验时继续观察。

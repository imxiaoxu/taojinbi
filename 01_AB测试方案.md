# Level 2 Agent 三组 A/B 测试方案

## 1. 实验目的与估计对象

**业务问题：**当用户在 `standard` 模式的 Level 2 发生失败时，动态 Agent 干预能否比固定规则和现有流程更有效地降低流失、提升关卡通过和游戏完成？

**主要估计对象：**在所有符合入组条件的用户中，Agent 相对 Holdout 的意向治疗效应（ITT）；同时比较 Agent 相对固定规则的增量效果。分析时以用户首次入组为起点，不因是否接受引导、是否切换模式而剔除用户。

## 2. 入组与随机化

| 项目 | 规则 |
|---|---|
| 入组事件 | `level_fail` 且 `game_variant=standard`、`level_id=2` |
| 触发时机 | 用户首次满足入组条件后，先完成实验分桶，再执行任一策略 |
| 排除 | 测试/风控账号、未成年或受奖励限制用户、核心字段缺失、当日已进入其他冲突实验用户 |
| 分桶单位 | `user_id_hash`；用户整个实验期固定分桶 |
| 分层随机 | `is_new_user`、`device_os`、`rules_viewed_before_fail` 三个分层内 1:1:1 随机 |
| 实验期 | 先完成最小样本量；7 日指标需在最后一位入组后完整观察 7 天 |
| 干扰控制 | 同一 session 仅允许一次主干预；后续仅展示静态提示，不跨组迁移 |

## 3. 分组与策略

| 组别 | 分流值 | 策略 |
|---|---|---|
| Holdout | `holdout` | 保持当前流程，不新增教程、模式切换或降难度动作；仍记录全量事件 |
| 固定规则组 | `fixed_rule` | 第一次 Level 2 失败后展示关键教程；第二次同关失败后切 `guided_mode` |
| Agent 组 | `agent` | 读取规则查看、同关失败次数、历史完成、设备性能、Memory 与频控，输出可变计划：教程、Guided Mode、降难度或静态兜底 |

**禁止事项：**Agent 组不得同时无条件叠加金币、优惠券等强激励；首期只验证游戏体验干预，避免将奖励因素混入关卡效果。

## 4. 指标与口径

### 4.1 主指标

| 指标 | 用户级口径 | 方向 |
|---|---|---|
| `level_2_pass_rate` | 入组后同一游戏 session 内出现 `level_complete(level_id=2)` 的用户 / 入组用户 | 越高越好 |
| `game_completion_rate` | 入组后 24h 内出现 `game_complete` 的用户 / 入组用户 | 越高越好 |

### 4.2 次指标

| 指标 | 用户级口径 |
|---|---|
| `level_2_immediate_exit_rate` | 入组后 10 分钟内出现 `game_exit(exit_point=2)` 的用户 / 入组用户 |
| `retention_7d` | 入组后第 7 天是否活跃 |
| `add_to_cart_7d` | 入组后 7 天内是否发生 `add_to_cart` |
| `order_7d` | 入组后 7 天内是否发生 `order_placed` |

### 4.3 护栏指标

| 类别 | 指标 | 预警阈值 |
|---|---|---|
| 成本 | 单入组用户奖励成本、单通过增量成本 | 首期不发券/金币；若启用奖励，超过预算 10% 即预警 |
| 体验 | `level_2_immediate_exit_rate`、负反馈率 | 任一实验组较 Holdout 恶化超过 3pp |
| 稳定性 | Dify P95、工具失败率、前端展示失败率 | Dify P95 > 1.5s；工具失败率 > 1%；展示失败率 > 0.5% |
| 频控 | 单 session 干预次数、重复执行率 | 干预次数 > 1 或重复执行率 > 0 |
| 数据 | SRM、关键事件完整率 | SRM p < 0.01；关键事件完整率 < 99% |

## 5. 最小样本量与成功标准

以历史 `standard` Level 2 基线通过率 `49.15%` 为起点。主分析包含 Agent vs Holdout、Agent vs 固定规则两次主要比较，因此每次比较使用双侧 `alpha=0.025`（Bonferroni 校正）、`power=80%`。

| 可检测绝对提升（MDE） | 每组所需完整用户数 | 三组总入组用户数 |
|---:|---:|---:|
| 3pp | 5,279 | 15,837 |
| 5pp（首期推荐） | 1,898 | 5,694 |
| 8pp | 739 | 2,217 |

**首期成功标准：**

1. Agent vs Holdout 的 `level_2_pass_rate` 提升至少 5pp，双侧检验 `p < 0.025`，95% CI 下限大于 0。
2. Agent vs 固定规则的 `level_2_pass_rate` 不低于固定规则，且 `game_completion_rate` 为正向；若其差异未达显著，则结论为“Agent 未证实优于固定规则”。
3. 所有护栏未触发停止阈值。
4. 分流、事件、执行日志均通过数据质量审计。

## 6. 停止、暂停与分析规则

| 类型 | 条件 | 动作 |
|---|---|---|
| 立即停止 | 风控/合规事故、重复发奖、核心游戏不可用 | 关闭 Agent 与固定规则动作，保留日志，执行回滚 |
| 暂停排查 | SRM p < 0.01、关键埋点缺失、Dify 或工具失败率超阈值 | 停止扩量；修复后重新做分流与数据检查 |
| 护栏暂停 | 体验指标相对 Holdout 恶化 > 3pp，且连续两日成立 | 降低流量并人工复核策略 |
| 正式分析 | 满足最小样本量且完成观察窗 | ITT 为主；按预设分层做异质性分析 |

不允许按日反复查看显著性后提前宣布成功。每日监控仅用于健康检查；业务结论在达到样本量和观察窗后一次性输出。

## 7. 数据质量与可解释性要求

1. 每日输出三组 `n`、分层分布和 SRM 检验；预期比例为 1:1:1。
2. `rules_viewed_before_fail` 必须来自失败事件前的规则查看记录，不能使用失败后发生的 `rule_view`。
3. `level_2_pass_rate` 的分母为入组用户，不以实际接受干预者为分母。
4. Agent 日志必须记录 `workflow_version`、`feature_version`、`decision_id`、`plan_json` 和每一步工具结果，以支持复盘。
